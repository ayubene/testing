### 等价类划分法
#### 等价划分
* 说明：在所有测试数据中，具有某种共同特征的数据集进行划分
* 分类
  + 有效等价类：满足需求的数据集合
  + 无效等价类：不满足需求的数据集合
* 步骤
  + 1.明确需求
  + 2.确定有效和无效等价类
  + 3.提取数据编写测试用例

  + 输入框

  + 单选复选框
* 典型代表：页面的输入框类测试
### 边界值分析法
* 用于解决边界位数限制问题
* 提示：
  + 有关范围限制，最多7条用例（暂时未优化）
  + 可以解决位数限制问题，不能解决类型问题（只有结合等价类才能解决）
#### 边界范围结点
* 选取正好等于，刚好大于，刚好小于，范围内的值作为测试数据
  + 上点（正好等于边界值的点）
  + 离点（离边界点最近的点，即刚好大于，刚好小于）
  + 内点（范围内的点）
* eg.取值[10,30]
  + 上点：10；30
  + 离点：9，11；29，31
  + 内点：20
#### 边界值法设计用例步骤
* 步骤
  + 1.明确需求
  + 2.确定有效和无效等价类（这里只需要考虑类型即可）
  + 3.确定边界范围值（这里列出来了所有的位数）
  + 4.提取数据编写测试用例

* 将7个点优化5个点

  + 离点：开内闭外（开区间选择内部离点，闭区间选择外部离点）
  + 内点（必选）
* eg.取值[10,30）
  + 上点：10；30
  + 离点：9；29
  + 内点：20
#### 使用场景
* 在等价类的基础上针对有边界范围的测试数据（重点关注边界）
* 常见词语描述：大小，长度，范围，尺寸，重量，最多，最少等
* 典型代表：有边界范围的输入框类测试

### 判定表法
#### 使用场景
* 使用在输入条件各种组合对输出结果有影响的情况
* 等价类划分法和边界值法主要关注单个输入类条件的测试
#### 定义及组成部分
* 定义：是一种以表格形式表达多条件逻辑判断的工具
* 组成：
  + 条件桩：列出问题中的所有条件，次序无关紧要
  + 动作桩：列出问题中可能采取的操作，次序无关紧要
  + 条件项：列出条件对应取值，所有可能情况下的真假值
  + 动作项：列出条件项各种取值下应该采取的动作结果
* 规则
  + 判定表中贯穿条件项和动作项的一列就是一条规则
  + 假设有n个条件，每个条件的取值有两个，则共有2的n次方个规则
#### 判定表法设计用例步骤
* 步骤
  + 1.明确需求
  + 2.画出判定表
    1) 列出条件桩和动作桩
    2) 填写条件项，对条件进行全组合
    3) 根据条件项组合确定动作项
    4) 简化、合并相似规则（有相同动作）
  + 4.提取数据编写测试用例
#### 案例2-5
* 需求：
* 要求：
  + 大于500，未过期，发批准单，提货单
  + 大于500，过期，不发批准单，提货单
  + 小于等于500，过期或者未过期，发批准单，提货单
  + 过期，无论多少钱，都发通知单
* [案例2-5测试用例](./案例2.xlsx)
#### 使用场景
* 有多个输入条件和输出结果，输入条件之间有组合关系，输入条件和输出结果之间有依赖（制约）关系
* 判定表一般适合条件组合较少的情况（4个条件以下）
* 如果条件超过4个，就不适合使用判定表法，应该采用正交法解决

### 场景法
#### 流程图
* 使用标准图形和箭头来表达程序或业务的走向
* 用于梳理流程明确相关信息；用于设计业务用例
* 流程图制作工具 
  + [网页版流程图制作](processon.com)
  + windows工具：visio
  + 其他工具：Excel

#### 场景法介绍
* 又叫流程图法，通过流程图描述用户的使用场景，通过覆盖流程路径来设计测试用例
* 意义
  + 用户角度：用户平时使用的不适单个功能，而是多个功能组合使用
  + 测试人员角度：平时测试都是单个功能测试，容易忽略多个功能的组合测试

### 错误推荐法
#### 介绍
* 定义：通过经验推测系统可能出现的问题
* 应用场景：当项目用例执行完毕，bug修复完成，离上线还有一段时间，可使用错误修复法复测主要业务或测试未覆盖的功能
